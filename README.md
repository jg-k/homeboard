# Homeboard

A Rails app for climbers with a home spray wall who want to centralize all their climbing and related training data in one place. Build problems on your wall and track sends; log outdoor crag sessions; record conditioning exercises and metrics; sync your system board ticks (Kilter / Tension / MoonBoard) via [Boardsesh](https://www.boardsesh.com/); and watch it all add up week by week.

Live at [homeboard.zone](https://homeboard.zone).

Hobby project — contributions welcome.

## What it does

- **Home boards** — register one or more home walls, upload a layout image, mark hold positions, create problems and grade them.
- **Activity log** — one timeline for board climbs, conditioning exercises, gym sessions, outdoor ascents, and hikes. GitHub-style activity grid with a color-by-intensity heatmap.
- **Conditioning** — define exercise types (with units, reps, optional added weight), log sets, set targets, see a progression chart per exercise.
- **Stats** — per-week climbing volume across home board / system board / indoor / outdoor.
- **Buddies** — follow other users to see their activity.
- **Imports** — pull outdoor ascents from theCrag or UKC; pull system board ticks (Kilter / Tension / MoonBoard) from [Boardsesh](https://www.boardsesh.com/).

## Stack

Standard Rails, plus Phlex for view components.

- Rails + SQLite + Hotwire (Turbo + Stimulus) + Solis queue and cach
- pure CSS
- import maps
- Phlex for view components (for experimentation...)
- Devise for authentication
- Kamal for deployment

## Setup

```bash
git clone https://github.com/jg-k/homeboard.git
cd homeboard
bin/setup            # bundles, prepares db, seeds, sets up master.key
bin/dev              # boots the server on http://localhost:3000
```

The seeds create a couple of test users with throwaway emails and a sample board. Log in as `bob@bob.com` / `password` to poke around (see `db/seeds.rb`).

The login page exposes an email/password form **only in development** (gated by `Rails.env.development?` in `app/views/devise/sessions/new.html.erb`). In production, the only login paths are the Google and Microsoft OAuth buttons — Devise's `:database_authenticatable` is enabled at the model level, but no UI surfaces it.

### Required configuration

Most things work out of the box. Two optional pieces:

**Environment**

| Var | Purpose |
| --- | --- |
| `RAILS_MASTER_KEY` | Generated by Rails on first `bin/setup` and written to `config/master.key`. Don't commit it. Required to read encrypted credentials. |

**Encrypted credentials** — edit with `bin/rails credentials:edit`. Required if you want OAuth to work in your own deploy (it's the only auth path in production).

```yaml
google_oauth2:
  client_id: ...
  client_secret: ...

microsoft:
  client_id: ...
  client_secret: ...
```

Integrations (theCrag, Boardsesh) take user-provided tokens entered through the Settings UI; nothing to configure server-side.
